Joomla!'s Backwards Compatability Promise from Joomla 3.3

  • Literally nothing can be broken
    • Whether in an extension or core library

But does this help

  • Let's introduce a hypothetical interface called JTableInterface which JTable should implement
  • Let's make JTable implement it
  • Now anybody who has E_STRICT errors in JTable will get an E_FATAL error
  • Technically this is a b/c break as the site would actually work with an E_STRICT and error reporting turned down

Who thinks this interface should be allowed to be put in Joomla! core?

Not quite so hypothetical!

  • This actually happened in Joomla 3.3 and allowed us to put tags and content history integration into F0F (and FOF packaged in core from Joomla! 3.4)
    • Not quite so clear cut!

More or less clear?: Background

  • So now lets take another example. With the CMSs JApplication layer you can create "clients"
  • You'd be familiar with the main 2 - "site" and "administrator"
  • When you do JApplication->isSite() or isAdmin(), you are actually checking these clients.

More or less clear?: The principle

  • I've built this super thing for Joomla! 3.5 - OAuth1 Server
  • Requires new application client to run things (logically)
  • Can anyone see the problem yet?

Who recognises code like this?


$app = JFactory::getApplication();
if ($app->isAdmin())
{
	// Do something in the backend
}
else
{
	// We are in the frontend - do something else
}

Oh dear


$app = JFactory::getApplication();
if ($app->isAdmin())
{
	// Do something in the backend
}
else
{
	// We are in the frontend - do something else
	// Oh wait - or are we in the new OAuth1 layer? (ooops)
}

So who's "right"

  • From a technical standpoint of the code there is no limit on the number of clients allowed
  • The code for my OAuth1 server is techincally correct
  • But at the point where mass implementation kicks in this is clearly never going to make it into core

The black, the white and the grey

  • When we say we aren't going to break b/c what actually is and isn't backwards compatible?
    • What happens if someone in JED had already made a com_tags component?
    • What happens if my new class called JViewHtmlCms (new MVC) conflicts with a "new" MVC class someone else has built

The black, the white and the grey: Conclusion

  • At the end of the day there is no perfect definition of what is and isn't a b/c break
  • So has the new policy on backwards compatibility really helped us?
  • Does the policy even make sense in extensions
  • We need to be able to define what is and isn't in the API
    • @api in the doc block?
    • Interaces only?
